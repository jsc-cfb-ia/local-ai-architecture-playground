# Guía de demo

## Objetivo

La demo muestra que el proyecto no es solamente un chat conectado a un modelo.
Es una aplicación RAG local con responsabilidades separadas:

1. Python recupera conocimiento local.
2. El ranking decide qué fragmentos son relevantes.
3. Qwen3 4B redacta usando esos fragmentos.
4. La aplicación devuelve contenido y fuentes estructuradas.
5. Si Ollama falla, el retrieval continúa funcionando.

Duración recomendada: 5 a 7 minutos.

## Preparación

Abrir Ollama y comprobar el modelo:

```bash
ollama list
ollama ps
```

Ejecutar las pruebas antes de presentar:

```bash
python -m unittest discover -v
```

Calentar el modelo para reducir la espera inicial:

```bash
ollama run qwen3:4b "Reply only with: ready /no_think"
```

## Demo rápida

Desde la raíz del proyecto:

```bash
python -m app.demo
```

El comando utiliza `ollama`, `qwen3:4b` y esta pregunta:

```text
Explain how AWS Step Functions participates in an event-driven architecture.
```

## Qué aparece

### 1. Retrieval

Se muestran los dos chunks mejor puntuados:

```text
1. aws.txt | topic=event-driven | chunk=1 | score=...
2. step-functions.txt | topic=orchestration | chunk=1 | score=...
```

Explicación para la audiencia:

> Estos fragmentos no los eligió el LLM. Mi retriever local los seleccionó
> mediante un ranking explicable. Así puedo evaluar búsqueda y generación por
> separado.

### 2. Grounded response

Qwen recibe la pregunta y únicamente los chunks recuperados. La aplicación
muestra la respuesta, las fuentes y si intervino el modelo.

Explicación:

> Qwen aporta comprensión y redacción, pero las fuentes siguen bajo control de
> la aplicación. El modelo no consulta internet ni APIs externas.

### 3. Agent-ready output

La misma respuesta se representa como JSON:

```json
{
  "content": "Generated answer",
  "sources": ["aws.txt", "step-functions.txt"],
  "generated_by_model": true,
  "answer_mode": "grounded",
  "warning": ""
}
```

Explicación:

> Esta salida estructurada permite que, en la siguiente etapa, Codex, Claude u
> otro agente invoque el asistente como tool o mediante MCP sin analizar texto
> de terminal.

## Guion recomendado

### Minuto 1: problema

> Quiero aprender a construir asistentes de arquitectura sin APIs de pago y
> manteniendo control sobre los datos, el retrieval y el modelo.

### Minuto 2: conocimiento

Mostrar `knowledge/aws.txt` y `knowledge/step-functions.txt`.

> El conocimiento está fuera del código. Puedo ampliarlo, versionarlo y
> reemplazarlo sin modificar el modelo.

### Minutos 3 y 4: ejecutar

```bash
python -m app.demo
```

Explicar retrieval, respuesta y JSON mientras aparecen.

### Minuto 5: arquitectura

Mostrar el diagrama de `docs/architecture.md`.

> El CLI es solo una interfaz. El núcleo es `ArchitectureAssistant`, el modelo
> implementa `LanguageModel` y el retriever es independiente. Por eso puedo
> convertirlo después en API, skill o MCP.

### Minuto 6: resiliencia

Ejecutar sin generación:

```bash
python -m app.demo --mode retrieval
```

> Incluso sin el LLM, la aplicación sigue recuperando conocimiento. Si Ollama
> se detiene durante el modo completo, se devuelve este resultado como
> fallback junto con una advertencia.

## Demo interactiva

Para conversar y enseñar memoria contextual:

```bash
python -m app.main
```

Secuencia sugerida:

```text
status
What is AWS Step Functions?
How does it relate to SQS?
memory
clear context
exit
```

La segunda pregunta demuestra contexto conversacional. `memory` muestra
persistencia y `clear context` separa memoria histórica de la ventana reciente.

## Preguntas alternativas

```bash
python -m app.demo \
  --question "How can AWS AppSync provide a GraphQL API?"
```

```bash
python -m app.demo \
  --question "Which AWS services appear in event-driven architectures?"
```

La base de conocimiento actual es deliberadamente pequeña. No conviene
presentar preguntas sobre seguridad, costos o patrones detallados porque esos
datos todavía no existen en `knowledge/`.

Para demostrar la diferencia entre evidencia local y conocimiento general:

```bash
python -m app.demo --question "Explain idempotency."
```

La respuesta debe mostrar:

```text
Sources: none
Generated by model: True
Answer mode: general
```

Esto no es RAG fundamentado. Es una respuesta útil del modelo, etiquetada para
que el usuario o un agente pueda decidir si necesita evidencia adicional.

## Qué no afirmar

- No es todavía búsqueda semántica: el ranking actual es lexical.
- No valida automáticamente que cada frase generada esté soportada.
- No es todavía un agente autónomo ni un servidor MCP.
- No contiene conocimiento completo de AWS.

Presentar estos límites muestra la evolución real del sistema y deja claro cuál
es el siguiente experimento.

## Plan B

Si Ollama no responde durante la presentación:

```bash
python -m app.demo --mode retrieval
```

El resultado sigue demostrando chunking, clasificación, ranking, fuentes y
salida estructurada. El warning del modo Ollama también evidencia el fallback.
